Projeto 2: Detecção de Cliques Fraudulentos

By Raphael Serra de Oliveira

July 9, 2020

Foram analisados dados públicos da maior plataforma independente de serviço de big data da China que cobre mais de 70% dos dispositivos móveis ativos em todo o país. São processados 3 bilhões de cliques por dia, dos quais 90% são potencialmente fraudulentos. A partir dos dados históricos foi criado um modelo de classificação que avalia se um usuário fará o download de um aplicativo para celular depois de clicar em um anúncio.

Os detalhes de código desenvolvido você encontra no meu Github.

Introdução

A maior plataforma independente de serviço de big data da China, cobre mais de 70% dos dispositivos móveis ativos em todo o país. São processados 3 bilhões de cliques por dia, dos quais 90% são potencialmente fraudulentos. Com mais de 1 bilhão de dispositivos móveis inteligentes em uso ativo todos os meses, a China é o maior mercado móvel no mundo e, portanto, sofre com grandes volumes de tráfego fradulento.

A abordagem atual para evitar a fraude de cliques para desenvolvedores de aplicativos é medir a jornada do clique de um usuário e sinalizar endereços IP que produzem muitos cliques, mas nunca acabam instalando aplicativos.

O problema de negócio

O risco de fraude está em toda parte, mas para as empresas que anunciam on-line, a fraude de cliques pode ocorrer em um volume esmagador, resultando em dados de cliques enganosos e desperdício de dinheiro.

Objetivo do Trabalho

Criar um modelo de classificação que avaliará se um usuário fará o download de um aplicativo depois de clicar em um anúncio de aplicativo para celular baseado na sua jornada (ip, app, dispositivo, hora do clique e outros).

Linguagem e Ferramentas Utilizadas

Este trabalho foi realizado com a linguagem R e com a ferramenta RStudio.

Kaggle e Fonte de Dados

Os dados trabalhados nesse projeto são públicos. Eles foram retirados do site Kaggle. Este site é amplamente utilizado por cientistas de dados que querem colocar o seu conhecimento à prova. Foi utilizado o dataset train_sample, que possui menos registros, pois não estou utilizando nenhuma técnica de Big Data. Portanto, foi feita uma análise estatística inferencial.

Desenvolvimento

Primeiramente foram carregados o arquivo train_sample.csv na memória do computador.

Feature Selection (Seleção de Variáveis)

Após os dados serem carregados e analisados superficialmente, a primeira etapa para solução foi aplicar a Seleção de Variáveis. A partir das informações que já tinhamos foram criadas as seguintes colunas no dataset:

train_sample <- train_sample %>%
          mutate(wday = as.factor(weekdays(click_time, abbreviate=T))) %>%
          mutate(hour = hour(click_time)) %>%
          select(-c(click_time)) %>%
          add_count(ip, wday, hour) %>% rename("nip_day_h" = n) %>%
          add_count(ip, hour, channel) %>% rename("nip_h_chan" = n) %>%
          add_count(ip, hour, os) %>% rename("nip_h_osr" = n) %>%
          add_count(ip, hour, app) %>% rename("nip_h_app" = n) %>%
          add_count(ip, hour, device) %>% rename("nip_h_dev" = n) %>%
          select(-c(ip, attributed_time))

wday = Dia da semana em que o anúncio foi clicado

hour = hora do clique

nip_day_h = Número de cliques de um mesmo IP, no mesmo dia e na mesma hora

nip_h_chan = Número de cliques de um mesmo IP, no mesma hora e do mesmo canal de anúncio

nip_h_osr = Número de cliques de um mesmo IP, na mesma hora e de um mesmo OS

nip_h_app = Número de cliques de um mesmo IP, na mesma hora e no mesmo APP

nip_h_dev = Número de cliques de um mesmo IP, na mesma hora e no mesmo dispositivo

Esse processo é muito importante, pois conseguimos gerar novas informações a partir das informações disponíveis. Isso aumenta o nosso entendimento sobre o problema e nos permite gerar um modelo de predição ainda mais preciso.

Também foram removidas as variáveis IP e click_time, pois essas informações agora estão nas novas colunas, e attributed_time (data do download), pois o que nos interessa saber é apenas se houve o download.

Com essa análise percebemos que o dataset está altamente desbalanceado. Existem muito mais registros sobre cliques que não geraram download do que cliques que geraram.
Se treinarmos o nosso algoritmo com os dados desse jeito ele aprenderá muito mais sobre cliques que não resultam em download. Dessa forma estaríamos criando um modelo tendencioso e com uma taxa de erro maior.

Após essa conclusão apliquei uma técnica através de uma função da linguagem R que balanceia os dados.

balanced_train_sample <- SMOTE(is_attributed ~ ., as.data.frame(train_sample), k = 3, perc.over = 400, perc.under = 150)

Machine Learning

Dividindo dataset em treino e teste

set.seed(123)
smp_size <- floor(0.70 * nrow(balanced_train_sample))
train_ind <- sample(seq_len(nrow(balanced_train_sample)), size = smp_size)

train <- balanced_train_sample[train_ind, ]
test <- balanced_train_sample[-train_ind, ]

Analisando relevância das variáveis para o modelo preditivo

Aqui utilizei o algoritmo randomforest como ferramenta para averiguar a importância das variáveis para o modelo preditivo e em seguida foi feito um plot.

modelo_rf1 <- randomForest(is_attributed ~ ., data=train, importance=TRUE)
varImpPlot(modelo_rf1)

Com essa informação escolhi as 7 variáveis de maior relevância. Foi adotado o método Gini para a escolha das variáveis. Variáveis: app + channel + hour + device + os + nip_h_dev + nip_day_h.

Foram utilizados dois algoritmos de aprendizado de máquina: RandomForest(com e sem parametrização) e Supported Vector Machine(kernel linear e radial).

Motivação da escolha dos algoritmos

Floresta Aleatória (Random Forest) é um algoritmo de aprendizagem de máquina flexível e fácil de usar que produz excelentes resultados a maioria das vezes, mesmo sem ajuste de hiperparâmetros. referência.

O SVM (Supported Vector Machine) apresenta o benefício de que você pode capturar relacionamentos muito mais complexos entre seus pontos de dados sem precisar realizar transformações complexas. A desvantagem é que o tempo de treinamento é muito maior, pois é muito mais computacionalmente intensivo. referência.

Treinamento

modelo_rf1 <- randomForest(is_attributed ~ app + channel + hour + device + os + nip_h_dev + nip_day_h, data=train)

# Análise da performance do modelo
# Criando dataframe com valores observados historicamente e com os valores previstos pelo modelo de machine learning
score_model <- data.frame(observado = test$is_attributed,
                          previsto = predictions <- predict(modelo_rf1, test[,-5]))

confusionMatrix(score_model$observado, score_model$previsto)

modelo_rf2 <- randomForest(is_attributed ~ app + channel + hour + device + os + nip_h_dev + nip_day_h, data=train, ntree = 300, nodesize = 3)

score_model2 <- data.frame(observado = test$is_attributed,
                          previsto = predictions <- predict(modelo_rf2, test[,-5]))

confusionMatrix(score_model2$observado, score_model2$previsto)

# Testando outro algorítimo de classificação SVM

# Utilizando kernel linear
modelo_svm_linear = tune.svm(is_attributed ~ app + channel + hour + device + os + nip_h_dev + nip_day_h, data=train,kernel="linear")

# Performance do modelo
best.linear=modelo_svm_linear$best.model
best.test=predict(best.linear,newdata=test,type="class")
confusionMatrix(best.test,test$is_attributed)

# Utilizando kernel radial
modelo_svm_radial = tune.svm(is_attributed ~ app + channel + hour + device + os + nip_h_dev + nip_day_h, data=train,kernel="radial")

# Performance do modelo
best.radial=modelo_svm_radial$best.model
best.test=predict(best.radial,newdata=test,type="class")
confusionMatrix(best.test,test$is_attributed)

Criando curvas ROC

# Criando curvas ROC
# Gerando as classes de dados
class1 <- predict(modelo_rf2, newdata = test, type = 'prob')
class2 <- test$is_attributed

# Gerando a curva ROC
pred <- prediction(class1[,2], class2)
perf <- performance(pred, "tpr","fpr")
plot(perf, col = rainbow(10))

A curva ROC, é uma representação gráfica que ilustra o desempenho de um sistema classificador binário. Quanto mais próximo de 1 na parte de cima da curva, melhor.

Conclusão

O modelo alcançou uma média de acurácia de 95% permitindo a conclusão do projeto. As estapas de seleção de variáveis e de balanceamento do dataset foram de estrema importância para que o modelo preditivo tivesse uma boa taxa de acertos. Além disso, o algoritmo RandomForest se sobressaiu em relação ao SVM.

Com isso conseguimos criar um modelo preditivo capaz de auxiliar a empresa no entendimento real da performance de seu anúncio e também se protejendo melhor encontra a ação de bots que geram gastos e não trazem retorno.
